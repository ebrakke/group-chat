# Changes - February 11, 2025

## Bug Fix: Admin Signup Shown When Users Exist

### Problem
The frontend showed the "Create Admin" signup form on the main page (/) even when an admin account already existed. This happened because when there was no authentication token, the frontend would default to `isFirstUser = true` without actually checking if users existed in the database.

### Solution
1. **API Change**: Added new public endpoint `GET /api/v1/auth/has-users`
   - Returns `{ hasUsers: boolean }`
   - Public endpoint (no auth required)
   - Used during initial page load to determine if signup or login should be shown

2. **Frontend API**: Added `checkHasUsers()` function to `frontend/src/lib/api.ts`
   - Calls the new endpoint
   - On error, defaults to `true` (assumes users exist → shows login page)
   - This is the safer behavior vs. exposing admin signup on API errors

3. **Frontend Page**: Updated `frontend/src/routes/+page.svelte`
   - Now properly checks if users exist before showing admin signup
   - Flow: No token → Check has users → Show signup (if false) OR redirect to login (if true)
   - On API error, redirects to login page instead of showing signup

### Files Modified
- `api/src/routes/auth.ts` - Added `/auth/has-users` endpoint
- `frontend/src/lib/api.ts` - Added `checkHasUsers()` function
- `frontend/src/routes/+page.svelte` - Updated `onMount()` logic

## Dev Environment Setup

### Purpose
Separate dev/staging environment so we can test changes without touching production.

### Changes
1. **docker-compose.dev.yml**
   - Frontend: port 3002 (instead of 3000)
   - API: port 4002 (instead of 4000)
   - Relay: port 3336 (instead of 3334)
   - Blossom: port 3337 (instead of 3335)
   - Separate volumes: `dev-api-data`, `dev-relay-data`, `dev-blossom-data`
   - Dev-specific secrets (clearly labeled as not for production)

2. **Makefile**
   - `make dev` — Start dev stack (builds + runs on ports 3002/4002)
   - `make dev-down` — Stop dev stack
   - `make dev-reset` — Stop dev stack and wipe all volumes
   - `make prod` — Build and start production stack (ports 3000/4000)
   - `make prod-down` — Stop production stack
   - `make prod-reset` — Stop production stack and wipe volumes
   - `make test` — Run messaging verification against dev stack
   - `make test-prod` — Run messaging verification against prod stack
   - `make help` — Show all available commands

3. **test-messaging.js**
   - Automated test script for messaging functionality
   - Tests: signup, send message, fetch messages, verify message
   - Takes API URL as argument (defaults to dev stack)
   - Colorized output with clear pass/fail indicators
   - Checks if users exist first (won't break existing environments)

### Files Created
- `docker-compose.dev.yml`
- `Makefile`
- `test-messaging.js` (executable)

## Usage Examples

### Start fresh dev environment and test
```bash
make dev-reset    # Clean slate
make dev          # Start dev stack
# Wait for containers to be ready
make test         # Run automated tests
```

### Work on dev, push to prod when ready
```bash
make dev          # Develop and test here
make test         # Verify changes
make dev-down     # Stop dev
make prod         # Start production
```

### Reset everything
```bash
make dev-reset    # Wipe dev data
make prod-reset   # Wipe prod data
```
